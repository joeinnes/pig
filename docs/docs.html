command reference

All three commands.

Complete reference for every pig command, flag, and config option.

pig [OPTIONS]

Launch an interactive grooming session. pig scans your configured paths, groups packages by name across all projects, and guides you through each conflict one at a time.

For each package that appears at multiple versions, you choose a resolution strategy. Approved changes are queued and written atomically at the end of the session. A confirmation prompt lets you review or discard the full queue before anything is touched.

OptionDescription
--package <name> Focus the session on a single package; all others are skipped
--strategy <name> Apply a named strategy to every package without interactive prompting. One of: lowest, most-acceptable, project, latest, specific
--dry-run Print proposed package.json changes without writing any files or running pnpm install
--validate <cmd> Override the validation hook for this session only. The command is run after each project's pnpm install; a non-zero exit reverts that project's changes
--no-validate Skip all validation hooks for this session
--paths <a,b,...> Override the configured scanPaths for this session; comma-separated glob patterns
# interactive session, all configured paths $ pig # focus on a single package with pre-selected strategy $ pig --package react --strategy latest # dry run — see what would change, write nothing $ pig --dry-run

pig scan [OPTIONS]

Scan all configured paths and print a version-conflict report. No files are modified. Exits 0 when the report is complete regardless of whether conflicts were found.

For each package that appears at more than one version, pig prints the versions in use and which projects reference each one. Unique packages (only one version across all projects) are summarised in a single line.

OptionDescription
--paths <a,b,...> Override the configured scanPaths for this run; comma-separated glob patterns
--dry-run Alias accepted; has no additional effect in scan mode (scan is already read-only)
$ pig scan Scanning 3 paths... react 2 versions 18.3.1 apps/dashboard, apps/admin 17.0.2 packages/legacy-widget lodash 2 versions 4.17.21 apps/dashboard 4.16.0 packages/utils 47 packages at a single version. 2 conflicts found.

pig store

Print a summary of the local pnpm content-addressable store: its path, total size on disk, and number of stored files.

The store path is resolved via pnpm store path unless overridden. Useful for a quick sanity-check before running pnpm store prune.

$ pig store Store path: /Users/joe/.local/share/pnpm/store/v10 Size: 4.2 GB Files: 312,847

Strategies

A strategy is how pig decides which version to pin for a conflicting package. Strategies can be applied interactively (you choose per-package) or globally via --strategy <name>.

NameHow it worksNetwork?
lowest Pin the lowest semver version currently in use across all projects. Safest for compatibility. No
most-acceptable Pick the version whose semver range is already satisfied by the most projects. Minimises the number of package.json edits needed. No
project Use the version installed in a specific project you choose. Useful when one project is the source of truth. No
latest Fetch dist-tags.latest from the npm registry and upgrade all projects to that version. Yes
specific Browse all published versions from the registry and pick one interactively. Versions are sorted newest-first. Yes
When most-acceptable results in a tie (two versions satisfy an equal number of ranges), pig prompts you to break the tie interactively.

Configuration

pig reads configuration from two places: a global user config at ~/.pig/config.json and an optional per-project override file at <projectRoot>/.pig.json.

Per-project settings take precedence over the global config for that project only. All fields are optional.

~/.pig/config.json — global config

Created automatically on first run if absent. Controls which project trees pig scans, which paths to ignore, and how to reach the npm registry.

FieldTypeDescription
scanPaths string[] Glob patterns or absolute paths to scan for pnpm-lock.yaml files. Supports ~/ expansion. Required — pig exits with an error if empty.
ignore string[] Glob patterns to exclude from scanning. Defaults to ["**/node_modules/**"].
validate string Shell command run after each project's pnpm install. A non-zero exit reverts that project's changes. Example: "pnpm test".
registryUrl string npm-compatible registry URL for latest and specific strategies. Defaults to "https://registry.npmjs.org".
{ "scanPaths": [ "~/WebDev/work", "~/WebDev/personal" ], "ignore": ["**/node_modules/**", "**/dist/**"], "validate": "pnpm run typecheck", "registryUrl": "https://registry.npmjs.org" }

.pig.json — per-project override

Add a .pig.json to the root of any project (next to its pnpm-lock.yaml) to override the global validation hook for that project only.

If a project's post-install checks differ from your global default — for example a project that needs a lighter-weight check — set it here rather than adjusting the global config.

FieldTypeDescription
validate string Shell command to run after pnpm install for this project. Overrides the global validate setting. Set to an empty string to disable validation for this project.
{ // lighter check for this project only "validate": "pnpm run build --dry-run" }
