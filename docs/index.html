Every monorepo sprouts
its own version of everything.

You work across ten projects. They all use React, lodash, TypeScript, and a dozen shared utilities. But each project evolves independently, pinned its dependencies at different times, and now they live in a zoo of conflicting versions.

The pnpm store deduplicates within a single install — but it cannot deduplicate across projects that never agreed on a version. Your store grows. Your brains hurt. Nobody wins.

🗂️

Scattered lockfiles

Every project has its own pnpm-lock.yaml. Without a global view, drift is invisible.

🔀

Silent version splits

lodash@4.16 and lodash@4.17 coexist quietly in the store. Neither project knows about the other.

📦

Wasted disk

The content-addressed store cannot reclaim space until all references to a version are gone.

🐛

Subtle bugs

Two projects sharing a package at different minor versions can behave differently in confusing ways.

Scan. Detect. Align. Done.

pig works in four stages. The first two are always free; the last two only happen when you choose to apply changes.

🔍

1 — Scan

Walk configured scan paths, find every pnpm-lock.yaml, parse all resolved packages.

🗺️

2 — Detect

Build a version group map. Surface only packages where two or more distinct versions are resolved.

🎯

3 — Strategise

Choose an alignment target per package: lowest, most-acceptable, a specific project, latest from registry, or a hand-picked version.

⚙️

4 — Apply

Update each package.json range, run pnpm install, optionally validate with a test command. Reverts on failure.

Three commands. Deliberately focused.

pig scan — read-only audit

Get a quick summary of version drift without changing anything. Safe to run any time.

$ pig scan --paths ~/work,~/personal

pig — interactive grooming session

The main event. pig presents each drifting package one at a time and lets you choose an alignment strategy. Changes are queued, reviewed, then applied together.

Five strategies for every situation.

pig never makes a unilateral decision. You choose how to resolve each package.

Strategy Target chosen Network Best for
lowest Lowest version in use No Maximum compatibility
most-acceptable Version satisfying most existing ranges No Minimise range bumps needed
project Version used by a chosen project No Anchoring to a reference monorepo
latest dist-tags.latest from registry Yes Staying current
specific Hand-picked from full version list Yes Pinning to a known-good version

Ready to wrangle your herd?

pig requires Node 22+ and pnpm installed globally.

$ pnpm add -g pig-cli
# or
$ npm install -g pig-cli

Configure scan paths

Create ~/.pig/config.json:

{
  "scanPaths": ["~/work", "~/personal"],
  "ignore":    ["**/node_modules/**"],
  "validate":  "pnpm test --run"
}
